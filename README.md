 # discord-code-reviewer


  Этот проект поднимает бота, который принимает команду !review, забирает код из текста сообщения и из прикреплённых файлов, очищает вход от типовых секретов, ограничивает
  объём данных и отправляет подготовленный запрос в Дипсик для инженерного разбора. Ответ возвращается обратно в канал как обычное сообщение, а если текст получился слишком
  длинным, бот отправляет файл с результатом.

  Служба рассчитана на простой сценарий: пользователь приносит небольшой фрагмент программы, бот даёт краткое ревью на русском языке, указывает сильные стороны, проблемы по
  приоритету, рекомендации, идеи для проверок и итоговый вердикт. Если ключ Дипсика не задан, проект не падает: вместо внешнего вызова включается локальная заглушка, чтобы
  можно было проверять поведение команды, ограничения и ответы без доступа к внешней модели.

  Бот не хранит долговременную историю ревью, не тянет код из репозиториев и не обращается к Гитхабу во время обычной работы. Его задача уже, чем у полноформатной ревью-
  системы: принять текст, безопасно сократить его до разумного объёма, не допустить перегрузки и вернуть понятный ответ в рамках ограничений Дискорда.

  ## Как подключить бота к серверу Дискорда

  1. Создайте приложение бота в панели разработчика Дискорда.
  2. Включите для бота привилегированный доступ к содержимому сообщений. Без этого бот не увидит текст после !review.
  3. Создайте токен бота и сохраните его отдельно. В коде проекта токен читается только из переменных окружения.
  4. Откройте раздел выдачи прав и сформируйте ссылку приглашения.
  5. Для работы достаточно прав на просмотр каналов, чтение истории сообщений, отправку сообщений и прикрепление файлов.
  6. Перейдите по ссылке приглашения, выберите нужный сервер и подтвердите добавление.
  7. После запуска проверьте команду в любом доступном текстовом канале:

  !review async def add(a, b): return a + b

  Если бот добавлен правильно, он покажет признак набора ответа, выполнит проверку и вернёт сообщение с ревью.

  ## Что нужно задать перед запуском

  Обязательна только одна переменная:

  DISCORD_TOKEN=

  Для работы через Дипсик обычно задают ещё и эти значения:

  DEEPSEEK_API_KEY=
  DEEPSEEK_ENDPOINT=https://api.deepseek.com/chat/completions
  DEEPSEEK_MODEL=deepseek-chat
  DEEPSEEK_TIMEOUT_SECONDS=60
  DEEPSEEK_MAX_RETRIES=3

  Параметры поведения самого бота:

  COMMAND_PREFIX=!
  REVIEW_COOLDOWN_SECONDS=300
  MAX_CONCURRENT_REVIEWS=3
  MAX_INPUT_CHARS=12000
  MAX_ATTACHMENT_BYTES=262144
  LOG_LEVEL=INFO
  LOG_FILE=

  Смысл настроек:

  - COMMAND_PREFIX задаёт префикс команд.
  - REVIEW_COOLDOWN_SECONDS вводит паузу между успешными ревью одного и того же пользователя.
  - MAX_CONCURRENT_REVIEWS определяет, сколько ревью можно выполнять одновременно на весь процесс.
  - MAX_INPUT_CHARS ограничивает суммарный размер текста, который уйдёт в Дипсик.
  - MAX_ATTACHMENT_BYTES ограничивает размер одного вложения.
  - DEEPSEEK_MAX_RETRIES задаёт число повторных попыток при временных ошибках внешнего сервиса.

  ## Как поднять службу локально

  Проект запускается обычным процессом на Пайтоне.

  pip install -r requirements.txt
  python main.py

  Точка входа находится в файле main.py. Она добавляет каталог src в путь импорта, загружает настройки из окружения, настраивает журналирование и передаёт управление
  объекту бота.

  Порядок локального старта такой:

  1. Подготовить окружение с токеном Дискорда.
  2. При необходимости добавить ключ Дипсика.
  3. Установить зависимости.
  4. Запустить python main.py.
  5. Проверить, что бот появился в сети и отвечает на !review.

  Если ключ Дипсика не задан, будет использован локальный имитатор ревью. Это удобно для проверки логики бота, но не заменяет настоящий внешний ответ модели.

  ## Как публиковать службу в облаке

  ### Размещение в Рендере

  В проекте уже есть файл render.yaml, который описывает рабочий процесс, а не веб-службу. Это важно: боту не нужен входящий сетевой порт, ему нужен постоянный фоновый
  процесс.

  Практический порядок:

  1. Создать новую службу рабочего типа.
  2. Указать репозиторий с проектом.
  3. Использовать сборку pip install -r requirements.txt.
  4. Использовать запуск python main.py.
  5. Перенести все переменные окружения в панель сервиса.

  Что происходит на бесплатном тарифе: у таких служб возможны усыпление при простое, холодный запуск после пробуждения и потеря локальных временных файлов при пересборке
  или перезапуске. Для бота это означает окно недоступности: пока процесс спит, он не читает события Дискорда и не отвечает на команды.

  ### Размещение в Рейлвей

  Порядок похожий:

  1. Создать проект.
  2. Подключить репозиторий.
  3. Указать команду сборки pip install -r requirements.txt.
  4. Указать команду запуска python main.py.
  5. Заполнить переменные окружения.

  На бесплатных или ограниченных по бюджету сценариях в Рейлвей надо отдельно следить за автоусыплением неактивных служб и за пределом месячного расхода. Если лимит
  расходов достигнут, платформа может остановить нагрузку целиком. Для бота это равносильно выключению обработчика событий.

  ### Размещение во Флай.ио

  На этой площадке бот обычно живёт как отдельная машина процесса.

  Общий порядок:

  1. Создать приложение.
  2. Подготовить конфигурацию запуска процесса.
  3. Передать секреты платформе.
  4. Выполнить выпуск новой версии.

  Для тестового режима у Флай.ио сейчас используется пробный период, а не классический постоянный бесплатный тариф для новых аккаунтов. В пробном режиме машины могут
  автоматически останавливаться через короткий простой, поэтому рассчитывать на непрерывную работу бота без оплаты нельзя. Для постоянно доступного бота нужен тариф без
  автоматического гашения процесса по условиям пробного режима.

  ## Почему сон бесплатных тарифов важен именно для этого проекта

  Этот бот не обслуживает входящие запросы по привычной схеме запрос-ответ через веб-сервер. Он живёт в постоянном соединении с Дискордом и ждёт события из цикла событий.
  Если платформа останавливает процесс, соединение обрывается. Дискорд не доставляет команды спящему процессу, а пользователь видит только молчание бота до момента нового
  запуска. Поэтому для рабочего использования лучше выбирать площадку и тариф, где фоновый процесс не отправляется в сон из-за бездействия.

  ## Где в проекте находится асинхронное ядро

  Цикл событий не создаётся вручную в коде проекта. Он поднимается внутри библиотеки discord.py в момент вызова bot.run(...) из файла src/discord_review_bot/app.py.

  Последовательность такая:

  1. main.py вызывает main().
  2. main() создаёт объект настроек и объект ReviewBot.
  3. ReviewBot.run(...) передаёт управление библиотеке discord.py.
  4. Библиотека создаёт и держит цикл событий asyncio.
  5. На этом цикле выполняются подключение к шлюзу Дискорда, чтение событий, вызовы команд, ожидание сетевых ответов и завершение фоновых задач.

  Иными словами, главный цикл живёт не в пользовательском коде, а внутри рантайма библиотеки. Код проекта только регистрирует асинхронные обработчики, которые будут вызваны
  на этом цикле.

  ## Как в коде появляются обработчики команд

  Регистрация команды происходит через декоратор @commands.command(name="review") в модуле src/discord_review_bot/commands/review.py. Но одной аннотации мало. Сам модуль
  оформлен как класс ReviewCog, а сам класс подключается в методе setup_hook() у ReviewBot.

  Фактическая цепочка регистрации выглядит так:

  1. Во время запуска discord.py вызывает setup_hook().
  2. Внутри setup_hook() создаётся сервис ревью и подключается ReviewCog.
  3. При добавлении ReviewCog библиотека просматривает методы класса.
  4. Метод, помеченный как команда, попадает в таблицу команд бота.
  5. Когда в сообщении появляется префикс ! и имя review, библиотека передаёт управление этому методу.

  Это означает, что команда не ищется вручную по строкам и не обрабатывается через самодельный разбор текста. Разбор префикса, имени команды и аргументов делает сама
  библиотека.

  ## Что здесь означает async def и чем оно отличается от обычной функции

  async def объявляет сопрограмму. Такая функция не выполняется до конца одним непрерывным куском. Она умеет останавливаться на await, отдавать управление циклу событий и
  продолжаться позже, когда внешний ввод-вывод завершится.

  Обычная функция def работает иначе:

  - вызывается немедленно;
  - занимает поток исполнения до возврата;
  - не умеет отдавать управление циклу событий посередине ожидания.

  Для этого проекта разница критична. Сетевые запросы к Дискорду, чтение вложений, ожидание ответа Дипсика и задержки повторных попыток должны быть неблокирующими. Если
  поставить туда обычные блокирующие вызовы, бот начнёт тормозить весь процесс, а не только один пользовательский запрос.

  ## Почему time.sleep(...) ломает такого бота

  time.sleep(...) усыпляет весь поток исполнения. Так как бот работает в одном процессе с циклом событий, блокирующий сон замораживает:

  - получение новых событий от Дискорда;
  - отправку ответов;
  - таймеры повторных попыток;
  - параллельные ревью других пользователей.

  asyncio.sleep(...) устроен иначе. Он не блокирует поток, а ставит задачу на паузу внутри цикла событий и позволяет выполнять в это время другие задачи. Именно поэтому в
  клиенте Дипсика повторные попытки и задержки построены через await asyncio.sleep(...).

  ## Полный путь выполнения команды !review

  Ниже описан реальный поток данных в этом проекте.

  1. Пользователь отправляет сообщение !review ... в канал.
  2. Шлюз Дискорда доставляет событие библиотеке discord.py.
  3. Библиотека создаёт объект контекста команды и вызывает метод review(...) внутри ReviewCog.
  4. Функция collect_review_content(...) забирает текст после команды.
  5. Если у сообщения есть вложения, берутся первые три файла.
  6. Для каждого файла проверяются размер и тип.
  7. Поддерживаемые текстовые файлы читаются асинхронно.
  8. Собранные куски объединяются в один текст.
  9. Если общий объём превышает MAX_INPUT_CHARS, хвост отрезается.
  10. Если вход пустой, вызывается пользовательская ошибка и бот отвечает пояснением.
  11. Бот включает признак набора ответа через context.typing().
  12. Управление передаётся в ReviewService.create_review(...).
  13. Сначала срабатывает пользовательский ограничитель частоты ReviewRateLimiter.start(user_id).
  14. Если у пользователя уже идёт активное ревью, новый запуск сразу запрещается.
  15. Если прошлое успешное ревью было слишком недавно, бот возвращает время ожидания.
  16. После этого запрос ждёт входа в глобальный семафор asyncio.Semaphore.
  17. Когда слот освобождается, текст проходит через очистку типовых секретов.
  18. Затем текст ещё раз ограничивается по длине.
  19. Формируется итоговый запрос к модели с инструкцией отвечать только по-русски и не повторять секреты целиком.
  20. Сервис Дипсика получает запрос через aiohttp.
  21. Если внешний сервис отвечает временной ошибкой или кодом ограничения частоты, выполняется повторная попытка с растущей задержкой.
  22. После успешного ответа результат возвращается в ReviewCog.
  23. Если итоговое сообщение помещается в лимит Дискорда, бот отправляет его в канал.
  24. Если не помещается, бот создаёт файл review.md и отправляет его вложением.
  25. В блоке finally ограничитель снимает отметку активного пользователя и, при успехе, записывает время завершения для паузы между ревью.

  ## Последовательность и параллельность внутри этого пути

  Не всё делается одновременно.

  Строго последовательно идут:

  - разбор входного сообщения;
  - проверка ограничителя по пользователю;
  - подготовка текста;
  - один внешний запрос в Дипсик;
  - отправка итогового ответа.

  Параллельность появляется только на уровне разных пользовательских задач. Каждая команда !review становится отдельной задачей в цикле событий. Пока одна задача ждёт
  чтение файла, сетевой ответ или паузу повтора, цикл может исполнять другие команды.

  Планирование задач делает сам asyncio. Каждый await создаёт точку, где текущая задача добровольно уступает исполнение. В этом проекте такие точки возникают при чтении
  вложений, отправке признака набора, входе в семафор, сетевом вызове к Дипсику, ожидании повторной попытки и отправке результата обратно в Дискорд.

  ## Что произойдёт при пяти одновременных запросах

  По умолчанию MAX_CONCURRENT_REVIEWS=3. Значит одновременно до внешнего вызова модели доходят не больше трёх задач.

  Если пять разных пользователей почти одновременно отправят !review, картина будет такой:

  1. Все пять событий попадут в цикл событий и будут разобраны.
  2. Все пять дойдут до пользовательского ограничителя.
  3. Если это пять разных пользователей и у них нет активного ревью, ограничитель их пропустит.
  4. Первые три задачи захватят семафор и начнут внешний запрос.
  5. Четвёртая и пятая задачи не пропадут. Они останутся ждать освобождения слота.
  6. Как только одна из первых трёх задач завершится, одна ожидающая задача продолжит работу.

  Следствие простое: бот работает параллельно, но не бесконтрольно. Параллельность ограничена семафором и дополнительным правилом "не больше одного активного ревью на
  пользователя".

  ## Как ограничивается нагрузка

  В проекте есть четыре уровня защиты.

  Первый уровень: ограничение по пользователю. Один и тот же пользователь не может держать несколько активных ревью одновременно.

  Второй уровень: пауза после успешного ревью. По умолчанию один пользователь может запускать новое ревью не чаще одного раза в триста секунд.

  Третий уровень: глобальный семафор процесса. Даже если много пользователей отправят запросы сразу, в Дипсик одновременно уйдёт не больше MAX_CONCURRENT_REVIEWS задач.

  Четвёртый уровень: ограничение входного объёма. Слишком большие файлы игнорируются, а слишком длинный суммарный текст обрезается.

  ## Ограничения внешних систем

  ### Ограничения Дискорда

  Для префиксной команды нужен доступ к содержимому сообщений. Если привилегированный доступ не включён, библиотека получит пустые поля сообщения и бот не сможет извлечь
  текст команды.

  Ограничения частоты у Дискорда нельзя жёстко прошивать числами в коде, потому что они зависят от маршрута и могут меняться. Официальная документация рекомендует опираться
  на заголовки ответа X-RateLimit-*, поле retry_after и код 429. Для приложений также действует глобальное ограничение на уровне приложения. В публичной документации
  Дискорда отдельно указано, что для одного приложения действует глобальный предел порядка пятидесяти запросов в секунду, но реальные правила всегда надо читать из ответов
  самого интерфейса.

  Для этого проекта практический риск ниже, чем у массовых ботов, потому что бот пишет мало: обычно один признак набора и одно итоговое сообщение на команду. Но при
  всплесках ошибок или бесконечных повторах можно уткнуться в ограничения и временно потерять возможность отправлять ответы.

  ### Ограничения Дипсика

  Проект обращается к одному внешнему методу создания ответа. В коде уже есть обработка кода 429, чтение Retry-After, повторные попытки и растущая пауза ожидания.

  По официальной документации Дипсика, ограничение параллельности считается на уровне учётной записи, а не только одного ключа. При превышении лимита сервис возвращает 429.
  На странице ограничений указаны отдельные пределы параллельных запросов для разных моделей. Это выше, чем локальный предел этого проекта, поэтому первым узким местом
  обычно станет именно MAX_CONCURRENT_REVIEWS, а не внутренняя планка Дипсика. Но если один и тот же ключ используется ещё и в других службах, общий лимит учётной записи
  может быть выбран быстрее.

  ### Ограничения Гитхаба

  В текущем коде прямого обращения к Гитхабу нет, поэтому эти лимиты не влияют на работу этого бота в его текущем виде.

  Если позже добавить загрузку файлов из репозитория, нужно помнить:

  - без аутентификации обычный предел составляет шестьдесят запросов в час на исходный адрес;
  - с личным токеном обычный предел составляет пять тысяч запросов в час;
  - у интерфейса есть и вторичные ограничения, которые могут сработать раньше обычного почасового лимита;
  - при превышении сервис возвращает 403 или 429, а время ожидания нужно брать из заголовков x-ratelimit-* и retry-after.

  Это важно заранее заложить в проект, если бот когда-либо начнёт ходить в репозитории автоматически.

  ## Стратегии повтора, растущей паузы и очереди

  В коде уже реализована базовая защита от кратковременных сбоев Дипсика:

  - при временной ошибке сети выполняется повтор;
  - при ответах семейства 5xx выполняется повтор;
  - при 429 используется пауза из Retry-After, а если заголовка нет, рассчитывается растущая задержка;
  - максимальное число повторов ограничено настройкой DEEPSEEK_MAX_RETRIES.

  Очередь как отдельная структура данных здесь не заведена, но роль очереди фактически выполняют задачи, ожидающие освобождения семафора. Они уже находятся в памяти
  процесса и продолжают ждать, пока одна из активных задач не завершится.

  Если нужно ещё более жёсткое управление нагрузкой, следующими логичными шагами будут:

  - отдельная очередь задач с явным лимитом длины;
  - ранний отказ при переполнении очереди;
  - раздельные лимиты для разных серверов;
  - журналирование времени ожидания в очереди.

  ## Что происходит с секретами и можно ли их утопить наружу

  Перед отправкой текста в Дипсик вызывается функция очистки типовых секретов. Она маскирует самые частые шаблоны ключей, токенов, паролей и некоторых известных форматов
  вроде префиксов личных ключей. Это снижает риск утечки, но не делает его нулевым.

  Важно понимать границы защиты:

  - маскировка работает по шаблонам, а не по полному пониманию смысла;
  - нестандартный секрет может пройти дальше, если не похож на известный шаблон;
  - содержимое вложения сначала читается локально, а затем уже очищается;
  - модель получает очищенный, но всё равно пользовательский код.

  Ответ на прямой вопрос такой: да, теоретически бот может отправить секреты в Дипсик, если они не попали под правила маскировки. Поэтому не стоит давать боту исходники с
  настоящими ключами, даже если в коде уже есть частичная очистка.

  ## Кто видит переменные окружения

  Переменные окружения не публикуются ботом в канал и не зашиваются в репозиторий, если владелец проекта работает правильно. Но они доступны тем, кто имеет доступ к среде
  выполнения:

  - владельцу локальной машины;
  - администраторам облачной площадки в пределах их модели доступа;
  - членам команды, которым выданы права на просмотр настроек сервиса;
  - процессу самого приложения.

  Если включить подробное журналирование и неосторожно писать значения в журнал, секреты могут попасть и туда. В текущем коде прямого логирования значений токенов нет, что
  правильно.

  ## Технический итог по устойчивости

  С точки зрения серверной реализации проект устроен просто и здраво:

  - цикл событий не блокируется намеренно;
  - пользовательские запросы режутся и по частоте, и по параллельности;
  - внешние ошибки не валят процесс мгновенно, а проходят через повторные попытки;
  - длинные ответы переводятся в файл, чтобы не упираться в размер одного сообщения Дискорда;
  - отсутствие ключа модели не ломает локальную проверку благодаря заглушке.

  Слабые места тоже понятны:

  - нет постоянной очереди вне памяти процесса;
  - нет распределённой координации между несколькими экземплярами бота;
  - нет полной гарантии удаления всех возможных секретов из пользовательского ввода;
  - при сне бесплатной площадки бот становится недоступен полностью, потому что вся его работа завязана на живой процесс и активное соединение с Дискордом.

  На 9 июня 2026 года официальные сведения по ограничениям и поведению площадок, использованные в этом описании, сверялись по документации Дискорда, Гитхаба, Дипсика,
  Рендера, Рейлвей и Флай.ио.
